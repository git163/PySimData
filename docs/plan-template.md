# Plan 文档模板

把 plan 保存到 `docs/` 目录下。命名推荐：

- 按主题命名（推荐用于长期文档）：`<主题>.md`，例如 `image-loader-redesign.md`
- 按日期命名（推荐用于一次性的计划）：`YYYY-MM-DD-<主题>.md`，例如 `2026-06-07-migrate-to-cpp20.md`

## 模板

```markdown
# <计划标题>

- 日期: YYYY-MM-DD
- 作者: <name>
- 状态: 草稿 / 评审中 / 已批准 / 已实施 / 已废弃
- 关联: <关联的 issue、文档或代码路径>

## 背景
为什么要做这件事？描述当前的问题、痛点、机会。

## 目标
- 目标 1
- 目标 2
- 非目标（明确列出不做什么，避免范围蔓延）

## 方案
详细描述方案。如果有多种方案，列出对比。

### 架构 / 流程
图或步骤说明（用 mermaid 或 ASCII 都可）。

### 关键设计点
- 关键点 1
- 关键点 2

### 接口 / 数据结构
必要时给出具体的接口签名或数据结构。

## 影响范围
- 哪些模块/文件会被影响
- 是否有兼容性影响（API、协议、存储格式）
- 是否有性能影响

## 风险与对策
- 风险 1: 对策
- 风险 2: 对策

## 替代方案
考虑了哪些其他方案？为什么没选？

## 实施步骤
- [ ] 步骤 1
- [ ] 步骤 2
- [ ] 步骤 3

## 测试计划
- 单元测试覆盖范围
- 集成 / 端到端测试
- 性能 / 回归测试（如适用）

## 引用
- 相关文档链接
- 相关 issue / PR
- 参考资料
```

## 写作建议

- 写给三个月后的自己看，假设对方没有这次讨论的上下文
- 目标 / 非目标一定要写清楚，避免范围蔓延
- 关键设计点要画图或举具体例子，纯文字很难讲清楚
- 风险不能只写"已评估"，要写具体风险和具体对策
